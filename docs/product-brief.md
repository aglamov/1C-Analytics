# Product Brief: 1C Analytics for iPhone and iPad

## Контекст

Проект `1C Analytics` - универсальное приложение для просмотра аналитических показателей из 1С на iPhone и iPad. Первый описанный сценарий связан с задачей SED-4884 и дашбордом "Образование".

Исходный документ: `/Users/aglamov/Downloads/Показатели_Образование.docx`.

## Пользовательская задача

Пользователь должен быстро открыть на iPhone или iPad актуальные показатели по образовательному контуру и понять:

- сколько всего обучающихся учитывается в 1С;
- как обучающиеся распределяются по укрупненным уровням образования;
- сколько обучающихся относится к РФ и иностранным гражданам;
- сколько обучающихся учится очно;
- на какую дату или срез построены данные.

## Первый дашборд

Название: `Образование`.

Источник данных: регистр сведений 1С `(вуз). Обучение`, срез последних.

Основные показатели из исходного документа:

| Показатель | Назначение | Базовая группировка |
| --- | --- | --- |
| Всего обучающихся | Общий объем контингента | Уровень образования |
| Всего обучающихся РФ и ИГ | Разделение контингента по гражданству | Уровень образования, гражданство |
| Обучающиеся, чел | Количество обучающихся в человеко-единицах | Уровень образования |
| Обучающиеся очно, чел | Количество обучающихся очной формы | Уровень образования, форма обучения |

## Нормализация уровней образования

В исходных запросах уровни образования приводятся к коротким группам:

| Группа | Значения |
| --- | --- |
| БАК | Бакалавриат |
| СПЕЦ | Специалитет |
| МАГ | Магистратура |
| БАК/СПЕЦ/МАГ | Объединенная группа для отдельных агрегатов |
| АСП | Аспирантура |
| ОРД | Ординатура, Ординатура+ |
| Прочие | Значения, не попавшие в основные группы |

## Сценарии приложения

1. Пользователь открывает приложение и видит состояние подключения к источнику данных.
2. Пользователь выбирает дашборд "Образование".
3. Приложение загружает показатели из API или мок-источника.
4. Пользователь видит сводные карточки с ключевыми значениями.
5. Пользователь открывает показатель и смотрит детализацию по группам.
6. Пользователь обновляет данные вручную, если нужен свежий срез.

## Требования к интерфейсу

- Главный экран должен быть рабочим дашбордом, а не лендингом.
- Карточки показателей должны быть компактными и читаемыми на iPhone.
- На iPad интерфейс должен использовать широкий экран: больше карточек в строке, боковая навигация и детализация рядом со списком.
- Один и тот же функциональный сценарий должен сохраняться на iPhone и iPad, меняется только компоновка.
- Для каждого показателя нужно показывать дату среза или время обновления.
- Детализация должна поддерживать сравнение групп без горизонтальной прокрутки там, где это возможно.
- Пустые состояния, ошибки сети и ошибки авторизации должны быть отдельными понятными состояниями UI.

## Адаптивность iPhone и iPad

Приложение должно проектироваться как `Universal`-приложение, а не как растянутый iPhone-интерфейс.

Для iPhone:

- вертикальная навигация через `NavigationStack`;
- одна колонка карточек на узких экранах;
- приоритет быстрых сводных значений;
- детализация открывается отдельным экраном;
- элементы управления должны быть удобны для одной руки.

Для iPad:

- навигация через `NavigationSplitView`;
- боковой список дашбордов или показателей;
- сетка карточек в 2-4 колонки в зависимости от ширины;
- детализация выбранного показателя отображается рядом с обзором;
- таблицы и сравнительные блоки используют ширину экрана, но не растягивают текст до неудобных строк.

Общие правила:

- использовать системные size classes и динамическую типографику;
- не задавать фиксированные размеры экранов;
- проверять портретную и альбомную ориентацию;
- поддерживать Dynamic Type без наложения текста;
- сохранять визуальную плотность деловой аналитики без декоративной перегрузки.

## Требования к данным

Минимальная модель показателя:

- идентификатор показателя;
- название;
- значение;
- единица измерения;
- дата или время среза;
- источник;
- строки детализации.

Минимальная модель строки детализации:

- название группы;
- значение;
- дополнительная размерность, если есть: гражданство, форма обучения или уровень образования;
- сортировка для стабильного отображения.

## Интеграция с 1С

Подготовлен HTTP API:

`https://sed2.rudn.ru/DGU_HTTP/hs/DGU_APP_Mobile_Client/analitycs/`

Для первого этапа нужно предусмотреть заменяемый слой данных:

- `MockAnalyticsProvider` для разработки интерфейса;
- `APIAnalyticsProvider` для подготовленного HTTP-интерфейса;
- единые модели, чтобы UI не зависел от формата ответа 1С.

Открытые вопросы:

- подтвердить фактическую схему ответа API;
- как будет устроена авторизация;
- нужно ли хранить токены на устройстве;
- какие роли пользователей должны видеть дашборд;
- требуется ли офлайн-кэш последнего успешного среза.

## Виды визуализаций

Клиент должен поддерживать виды графиков, которые приходят из API или назначаются в модели показателя:

| Вид | Назначение в приложении |
| --- | --- |
| `BarMark` | Вертикальная столбчатая диаграмма для сравнения групп |
| `BarMarkHorizon` | Горизонтальная столбчатая диаграмма для длинных подписей и рейтингов |
| `BarMarkStacking` | Составная столбчатая диаграмма для сравнения частей внутри группы |
| `SectorMarkInnerRadius` | Кольцевая диаграмма для структуры и долей |

В iOS-клиенте эти виды должны быть реализованы через Swift Charts. Внутреннее имя вида лучше сохранить близким к API, а конкретный SwiftUI-компонент выбирать через слой маппинга.
