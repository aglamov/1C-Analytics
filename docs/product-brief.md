# Product Brief: 1C Analytics for iPhone

## Контекст

Проект `1C Analytics` - мобильное приложение для просмотра аналитических показателей из 1С на iPhone. Первый описанный сценарий связан с задачей SED-4884 и дашбордом "Образование".

Исходный документ: `/Users/aglamov/Downloads/Показатели_Образование.docx`.

## Пользовательская задача

Пользователь должен быстро открыть на iPhone актуальные показатели по образовательному контуру и понять:

- сколько всего обучающихся учитывается в 1С;
- как обучающиеся распределяются по укрупненным уровням образования;
- сколько обучающихся относится к РФ и иностранным гражданам;
- сколько обучающихся учится очно;
- на какую дату или срез построены данные.

## Первый дашборд

Название: `Образование`.

Источник данных: регистр сведений 1С `(вуз). Обучение`, срез последних.

Основные показатели из исходного документа:

| Показатель | Назначение | Базовая группировка |
| --- | --- | --- |
| Всего обучающихся | Общий объем контингента | Уровень образования |
| Всего обучающихся РФ и ИГ | Разделение контингента по гражданству | Уровень образования, гражданство |
| Обучающиеся, чел | Количество обучающихся в человеко-единицах | Уровень образования |
| Обучающиеся очно, чел | Количество обучающихся очной формы | Уровень образования, форма обучения |

## Нормализация уровней образования

В исходных запросах уровни образования приводятся к коротким группам:

| Группа | Значения |
| --- | --- |
| БАК | Бакалавриат |
| СПЕЦ | Специалитет |
| МАГ | Магистратура |
| БАК/СПЕЦ/МАГ | Объединенная группа для отдельных агрегатов |
| АСП | Аспирантура |
| ОРД | Ординатура, Ординатура+ |
| Прочие | Значения, не попавшие в основные группы |

## Сценарии приложения

1. Пользователь открывает приложение и видит состояние подключения к источнику данных.
2. Пользователь выбирает дашборд "Образование".
3. Приложение загружает показатели из API или мок-источника.
4. Пользователь видит сводные карточки с ключевыми значениями.
5. Пользователь открывает показатель и смотрит детализацию по группам.
6. Пользователь обновляет данные вручную, если нужен свежий срез.

## Требования к интерфейсу

- Главный экран должен быть рабочим дашбордом, а не лендингом.
- Карточки показателей должны быть компактными и читаемыми на iPhone.
- Для каждого показателя нужно показывать дату среза или время обновления.
- Детализация должна поддерживать сравнение групп без горизонтальной прокрутки там, где это возможно.
- Пустые состояния, ошибки сети и ошибки авторизации должны быть отдельными понятными состояниями UI.

## Требования к данным

Минимальная модель показателя:

- идентификатор показателя;
- название;
- значение;
- единица измерения;
- дата или время среза;
- источник;
- строки детализации.

Минимальная модель строки детализации:

- название группы;
- значение;
- дополнительная размерность, если есть: гражданство, форма обучения или уровень образования;
- сортировка для стабильного отображения.

## Интеграция с 1С

Пока транспорт не зафиксирован. Для первого этапа нужно предусмотреть заменяемый слой данных:

- `MockAnalyticsProvider` для разработки интерфейса;
- `APIAnalyticsProvider` для будущего HTTP-интерфейса;
- единые модели, чтобы UI не зависел от формата ответа 1С.

Открытые вопросы:

- какой способ подключения будет использоваться: прямой HTTP-сервис 1С, промежуточный backend или выгрузка;
- как будет устроена авторизация;
- нужно ли хранить токены на устройстве;
- какие роли пользователей должны видеть дашборд;
- требуется ли офлайн-кэш последнего успешного среза.

## План реализации

1. Создать нативный iOS-проект.
2. Собрать навигационный каркас приложения.
3. Добавить мок-модели и мок-данные для дашборда "Образование".
4. Реализовать карточки показателей и экран детализации.
5. Выделить протокол источника данных.
6. Подключить реальный API, когда будет известен контракт интеграции.
7. Добавить обработку авторизации, ошибок и состояний загрузки.
